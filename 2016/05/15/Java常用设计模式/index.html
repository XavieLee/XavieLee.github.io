Java常用设计模式

设计模式
  • Java

策略模式

写在前面-为什么要学习设计模式？

  • 对于接触编程不久的我来说，编程的目的就是简简单单的实现我想实现的功能。再到接触到设计模式，我认为这样写代码真是浪费时间浪费精力，很简单的实现非得搞得很复杂，再后来随着软件开发经验的增加，我才开始明白我之前以为的简单仅仅是设计的简单，而现实中软件开发中，简单随意的设计让你的代码可读性大打折扣，复用性几乎不存在。设计模式就是将一类问题封装起来处理，将逻辑独立的代码封装出来，提高代码的复用性，可移植性。

策略模式详解

  • 模拟鸭子游戏的应用程序，要求：游戏中会出现各种颜色外形的鸭子，一边游泳戏水，一边呱呱叫。

第一种方法：

  • 直接编写出各种鸭子的类：MallardDuck//野鸭，RedheadDuck//红头鸭，各类有三个方法：
    quack()：叫的方法
    swim()：游水的方法
    display()：外形的方法

    第二种方法:

  1. 设计一个鸭子的超类（Superclass）,并让各种鸭子继承这个超类。

  2. 对于它的子类只需简单的继承就可以了，并实现自己的display()方法。

  3. 现在客户又提出了新的需求，想让鸭子飞起来。我们在超类中在加一个方法就可以了。

  1. 对于不能飞的鸭子，在子类中只需简单的覆盖。

  2. 其它会飞的鸭子不用覆盖。这样所有的继承这个超类的鸭子都会fly了。但是问题又出来了，客户又提出有的鸭子会飞，有的不能飞。

    休息思考：

  • 对于上面的设计，你可能发现一些弊端，如果超类有新的特性，子类都必须变动，这是我们开发最不喜欢看到的，一个类变让另一个类也跟着变，这样很显然的耦合了一起，既然耦合度太高，我们就要降低耦合度。

    第三种方法：

  1. 用接口改进.
    我们把容易引起变化的部分提取出来并封装之，来应付以后的变法。虽然代码加大了，但可用性提高了，耦合度也降低了。

  • 好处:
    这样已设计，我们的程序就降低了它们之间的耦合。
  • 弊处:
    Flyable和 Quackable接口一开始似乎还挺不错的，解决了问题（只有会飞到鸭子才实现 Flyable），但是Java接口不具有实现代码，所以实现接口无法达到代码的复用。

    总结:

  • 继承的优点:复用性提高。
  • 继承的弊处:耦合性高，一旦超类添加一个新方法,子类都继承,拥有此方法,
    若子类相当部分不实现此方法,则要进行大批量修改。继承时,子类就不可继承其它类了.
  • 接口的优点:解决了继承耦合性高的问题，可以继承或实现其它类或接口。
  • 接口的弊处:不能真正实现代码的复用。

strategy(策略模式)

设计原则：

  1. 找出应用中相同之处，且不容易发生变化的东西，把它们抽取到抽象类中，让子类去继承它们；
  2. 找出应用中可能需要变化之处，把它们独立出来，不要和那些不需要变化的代码混在一起。(important)
  3. 现在，为了要分开“变化和不变化的部分”，我们准备建立两组类（完全远离Duck类），一个是”fly”相关的，另一个是“quack”相关的，每一组类将实现各自的动作。比方说，我们可能有一个类实现“呱呱叫”，另一个类实现“吱吱叫”，还有一个类实现“安静”。

首先写两个接口,FlyBehavior(飞行行为)和QuackBehavior(叫的行为).
